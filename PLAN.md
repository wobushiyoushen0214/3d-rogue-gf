# 3D 国风修真肉鸽统一计划（结合旧玩法蓝图与当前模板结构）

## Summary
- 项目定位仍沿用之前的方向：`单人爽刷成长` 的 `国风修真题材 3D 肉鸽`，核心体验是 `低门槛移动操作 + 自动攻击 + 高频成长反馈 + 流派成型 + 长线重复开局`。
- 当前项目不是从零开始，而是已经有一个 `3D 可运行战斗模板`，包括 `角色移动`、`随机刷怪`、`RVO 动态避障`、`无限地图循环`、`相机跟随`、`暂停与继续`、`2D UI 跟随 3D`、`基础伤害表现`。
- 本计划的目标，是把之前偏产品设计的完整玩法蓝图，和当前代码目录、现有功能、实际实现顺序合并成一份可以直接执行的研发计划。
- 整体策略是：
  - 先把当前模板收口成 `稳定可玩的 3D 生存战斗基础版`。
  - 再在这个基础上扩展 `build、章节、无尽模式、局外成长`。
  - 不推翻已有结构，优先沿用 `view/game + managerGame + utils/RVO + data/povo` 的现有分层。

## 产品方向
### 1. 题材与定位
- 题材继续保持 `国风修真`，不走现代枪械或纯科幻方向。
- 战斗表现以 `御剑、符法、法宝、阵法、妖潮` 为核心包装。
- 画面形态以当前 `3D 场景模板` 为基础，后续通过 `技能特效、敌人类型、UI 包装` 逐步增强题材感。

### 2. 核心体验
- 输入层面：玩家主要只控制 `移动`，攻击和大部分技能 `自动释放`。
- 成长层面：通过 `击杀 -> 获得经验 -> 升级/强化 -> 流派形成 -> 更高压敌潮` 建立爽感。
- 反馈层面：强调 `怪物受击闪烁`、`伤害数字`、`怪物死亡`、`数量压制`、`技能成长` 的连续正反馈。
- 长线层面：中后期再引入 `章节挑战`、`无尽试炼`、`局外成长`、`法宝/道具/流派`。

### 3. 当前版本目标
- 当前版本不是直接做完整版内容，而是先完成一版 `3D 国风肉鸽战斗底盘`。
- 这版必须跑通：
  - `角色移动`
  - `相机跟随`
  - `随机生成怪物`
  - `RVO 动态避障`
  - `怪物受击闪红（白）`
  - `2D UI 跟随 3D 角色`
  - `无限地图移动`
  - `游戏暂停与开始`
  - `基础操作按钮`
  - `工具/武器/道具状态修改`

## 当前版本交付定义
### 1. VS1 首个垂直切片
- 这是当前阶段真正的第一条交付线，不是长期愿景。
- VS1 内容包定义为：
  - `1 个角色`
  - `1 张可循环地图`
  - `2 类怪物（近战 / 远程）`
  - `1 套基础自动攻击`
  - `1 套调试型属性修改入口`
  - `1 套暂停 / 继续 / 重开 / 游戏结束流程`
  - `1 局可持续 8-10 分钟的生存战斗`

### 2. VS1 玩家体验目标
- 玩家在 `60 秒内` 理解移动、自动攻击、杀怪反馈。
- 玩家在 `2-3 分钟内` 至少感受到 `1 次明显成长`。
- 玩家在 `6-8 分钟` 进入明显高压期，而不是只是在原地重复清怪。
- 整局体验必须更像一个“能玩”的战斗切片，而不是一组分散功能的演示。

### 3. VS1 完成定义
- 连续游玩 `10 分钟` 不出现阻断流程的错误。
- 地图循环、刷怪、RVO、受击、暂停、重开全部串成一条稳定链路。
- 即使暂时没有完整升级系统，也必须具备 `可观察的成长变化`，可以先通过调试型属性修改入口或简化版升级实现。
- 代码结构允许后续无缝接入 `升级三选一`、`精英/Boss`、`单局结算`。

## 完整玩法循环
### 1. 长循环
- 玩家从后续的 `主界面 / 洞府` 进入战斗。
- 开局前选择或配置：
  - `角色`
  - `初始术法 / 武器`
  - `道具 / 法宝`
  - `命格 / 初始强化倾向`
- 进入战斗后，通过 `移动生存 + 自动攻击 + 杀怪升级 + 选择强化` 持续成长。
- 单局结束后结算资源，用于后续扩展的 `局外成长`。
- 当前模板阶段先不实现完整大厅，但架构上要为这一套流程预留。

### 2. 单局循环
- `0-60 秒`
  - 低压开局，帮助玩家建立移动、攻击、击杀反馈。
- `1-3 分钟`
  - 刷怪频率提升，开始形成基础 build。
- `3-6 分钟`
  - 怪群更密集，避障、走位、输出效率开始重要。
- `6 分钟后`
  - 进入高压期，可衔接精英怪或 Boss。
- 当前模板阶段可先聚焦 `生存刷怪 + 无限地图 + 数值成长`，Boss 与章节机制放到后续版本。

## 结合当前代码结构的落地方案
### 1. 当前项目结构映射
- `assets/script/view/game/level.ts`

- `assets/script/view/game/MapDirTS.ts`
  - 地图块尺寸计算、世界定位、静态障碍注入 `RVO obstacle`。
- `assets/script/managerGame/MonsterManager.ts`
  - 怪物 prefab 管理、对象池复用、RVO agent 生命周期管理、敌人位置更新。
- `assets/script/managerGame/Monster.ts`
  - 怪物状态、受击、死亡、材质闪烁、碰撞处理。
- `assets/script/view/game/EnemyTS.ts`
  - 怪物 AI 行为层，负责近战/远程攻击判断。
- `assets/script/managerGame/Actor.ts`
  - 主角运动、朝向、受伤、碰撞、基础状态机。
- `assets/script/view/game/PlayerTs.ts`
  - 玩家输入转角色状态、自动索敌、自动发射投射物、升级成长。
- `assets/script/view/game/CameraTs.ts`
  - 相机跟随主角。
- `assets/script/view/game/UIGameTs.ts`
  - 暂停、继续、重开、游戏结束界面。
- `assets/script/view/game/UIJoyStick.ts`
  - 摇杆输入。
- `assets/script/view/game/UIMaterInfoTS.ts` + `assets/script/view/game/HurtNumber.ts`
  - 主角血条、伤害数字等 2D UI 跟随 3D 角色。
- `assets/script/utils/RVO/*`
  - 动态避障基础能力。
- `assets/script/data/povo/*`
  - 运行时角色、道具、投射物、关卡配置数据。

- 单局结束后有明确结算反馈，形成“开局 -> 成长 -> 压力 -> 结束”的完整闭环。

## 平台与性能预算
### 1. 目标平台
- 优先按 `微信小游戏` 的 3D 项目约束设计。
- 当前默认目标形态仍是 `竖屏单手操作`。
- 性能基线优先参考 `中端安卓设备`，不要只按编辑器表现判断可行性。

### 2. P0 性能预算
- 常态战斗帧率目标：`>= 45 FPS`
- 敌潮高压时最低可接受帧率：`>= 30 FPS`
- 同时活跃怪物目标：`20-30`
- 极限存活怪物上限：`40`
- 连续运行 `10 分钟`，不应出现持续恶化的卡顿。
- 场景连续 `重开 3 次` 后，不应出现明显的节点、agent、UI 残留增长。

### 3. P1 性能预算
- 接入升级、精英怪、更多特效后，仍需保持：
  - 常态战斗 `>= 45 FPS`
  - 高压阶段 `>= 30 FPS`
- 若 P1 内容导致预算失守，优先收缩：
  - 特效密度
  - 同屏怪物数
  - UI 跟随对象数量
  - 高频非池化对象创建

### 4. 工程约束
- 新增伤害数字、特效、怪物、UI 跟随节点时，优先考虑 `对象池`。
- `RVO`、地图刷新、刷怪逻辑都必须可节流，不能无限叠加更新成本。
- 不允许为了追求视觉效果破坏 P0/P1 的移动端性能预算。

## Test Plan
- 角色移动
  - 摇杆连续拖动 `3 分钟`，角色无方向错乱、无异常停滞。
- 相机跟随
  - 快速转向、暂停、重开后都能稳定跟随玩家。
- 随机刷怪
  - 怪物不会出生在玩家碰撞体内，不会出生到不可达障碍中。
- RVO 动态避障
  - `20-50` 只怪同时追击时，能绕开彼此和墙体，不出现大面积穿模、重叠、卡死。
- 无限地图移动
  - 主角持续朝一个方向移动至少 `5 分钟`，地图循环无明显缝隙、跳变、障碍错乱。
- 怪物受击闪红（白）
  - 连续命中时材质切换稳定，回收复用后不会残留错误材质。
- 2D UI 跟随 3D
  - 血条、伤害数字在相机移动和目标移动时位置准确，目标销毁后 UI 清理正常。
- 游戏暂停与开始
  - 暂停后怪物、角色、刷怪、特效、动画都停止；恢复后继续正常；游戏结束只允许进入重开流程。
- 工具状态修改
  - 运行中调整攻击、攻速、子弹数量、穿透、追踪后，战斗表现与 UI 显示一致。
- 单局成长
  - 后续接入升级后，玩家应能在 `2-3 分钟` 内感受到至少一次明显成长。
- 性能预算
  - `P0/P1` 均需要在目标测试设备上满足既定帧率和怪物数量预算，不能只在编辑器里通过。
- 内存与回收
  - 连续游玩 `10 分钟`、重开场景 `3 次` 后，不应出现明显的 agent、节点、UI 残留累积。
- 无报错要求
  - 主流程连续测试过程中，不应出现持续刷屏的控制台报错或关键逻辑异常。

## Risks
- `MapDirTS` 里障碍去重逻辑当前没有真正启用，地图循环后有重复 obstacle 的风险。
- `MonsterManager` 的 agent 生命周期与节点回收高度耦合，最容易出现“节点没了但 agent 还在”的问题。
- `CameraTs` 目前是直接赋值跟随，后续加平滑和震动时要注意不要破坏 UI 投影结果。
- `UIGameTs` 直接调用 `director.pause()`，若后续需要保留部分 UI 动画，就必须拆出独立更新策略。
- `UIJoyStick` 使用全局输入监听，后续按钮变多后容易与 UI 点击冲突。
- 当前模板更偏“技术底盘”，若不结合旧计划持续扩内容，容易停留在 Demo 状态。
- 若没有性能预算约束，`RVO + 无限地图 + 跟随 UI + 特效` 很容易在移动端叠成不可接受的开销。

## Assumptions
- 当前优先目标仍是 `先把 3D 模板收口，再逐步产品化`，不直接上完整大体量内容。
- 当前版本继续坚持 `低输入复杂度`，玩家以移动为主，不引入复杂连招。
- 当前先做 `单人`，不考虑多人、PVP、联机。
- 旧计划中的 `章节、无尽、洞府、命格、灵宠、法宝流派` 继续保留为中长期方向，但不在本轮一次性压进当前模板阶段。
